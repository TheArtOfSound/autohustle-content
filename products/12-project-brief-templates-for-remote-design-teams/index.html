[Clearly describe the primary goal of the project. Example: Redesign the mobile onboarding flow to increase sign-up conversion by 20%.]

Scope:

  • Included:
  • [LIST KEY DELIVERABLES, e.g., User flows, high-fidelity mockups, design system updates]
  • Excluded:
  • [LIST OUT-OF-SYNC WORK, e.g., Frontend development, copywriting, QA testing]

Success Metrics:

  • [KPI 1, e.g., 15% reduction in user drop-off]
  • [KPI 2, e.g., 90% usability task success rate in testing]
  • [KPI 3, e.g., Stakeholder approval by [DATE]]

Timeline:

  • Kickoff: [DATE]
  • Research Complete: [DATE]
  • Wireframes Delivered: [DATE]
  • High-Fidelity Designs: [DATE]
  • Final Handoff: [DATE]

Communication Plan:

  • Daily Standups: [TIME] via [VIDEO CALL PLATFORM], [FREQUENCY, e.g., Mon-Fri]
  • Weekly Sync: [DAY, TIME] with stakeholders via [PLATFORM]
  • Async Updates: Shared in [SLACK CHANNEL / NOTION PAGE / OTHER TOOL]

Tools & Access:

  • Design: [FIGMA / SKETCH / OTHER] – [LINK TO FILE]
  • Documentation: [NOTION / CONFLUENCE / OTHER] – [LINK]
  • Feedback: [INVISION / FIGMA COMMENTS / OTHER]

  • [NAME] – [ROLE] – [RESPONSIBILITY, e.g., Final sign-off]

Template 2: UX Research Brief

When to use: Before beginning user research to align remote researchers and designers on methodology, participants, and goals.

Project: [PROJECT NAME]

Research Lead: [RESEARCHER NAME]

Design Team: [LIST TEAM MEMBERS]

Stakeholders: [LIST NAMES + ROLES]

Research Goal:

[Explain what you want to learn. Example: Understand pain points users experience during checkout on mobile devices.]

Research Questions:

  1. [QUESTION 1, e.g., What causes users to abandon the cart?]
  2. [QUESTION 2, e.g., How do users perceive trust signals during payment?]
  3. [QUESTION 3, e.g., What information do users expect to see before purchasing?]

Methodology:

  • Type: [USER INTERVIEWS / SURVEYS / USABILITY TESTING / OTHER]
  • Format: [ASYNC VIDEO / LIVE VIDEO CALLS / IN-PERSON OBSERVATION]
  • Duration: [LENGTH PER SESSION]
  • Sample Size: [NUMBER] participants

Participant Criteria:

  • Demographics: [AGE RANGE, LOCATION, JOB ROLE, ETC.]
  • Behavior: [SPECIFIC USAGE PATTERNS, e.g., made a purchase in last 30 days]
  • Recruitment Source: [PLATFORM, e.g., UserTesting, internal panel, social media]

Timeline:

  • Recruitment: [START DATE] – [END DATE]
  • Sessions: [START DATE] – [END DATE]
  • Analysis & Insights: [DATE]
  • Reporting: [DATE]

Deliverables:

  • [RAW RECORDINGS / ANONYMIZED]
  • [TRANSCRIPTS]
  • [AFFINITY MAP / THEMATIC ANALYSIS]
  • [INSIGHTS DECK] – Shared via [LINK TO DOCUMENT]

Tools:

  • Recruitment: [PLATFORM]
  • Sessions: [ZOOM / LOOKBACK / OTHER]
  • Notes & Synthesis: [NOTION / MIRO / OTHER] – [LINK TO BOARD]

Ethics & Consent:

  • All participants will sign a consent form outlining data usage.
  • Recordings will be stored in [SECURE LOCATION] and deleted after [TIME PERIOD].

Next Steps:

  • [ACTION ITEM 1, e.g., Finalize screener survey]
  • [ACTION ITEM 2, e.g., Share calendar availability for session moderation]
  • [ACTION ITEM 3, e.g., Confirm stakeholder attendance for 2 test sessions]

Template 3: Design Sprint Brief

When to use: To run a time-boxed, remote design sprint focused on solving a specific problem.

Sprint Title: [PROBLEM TO SOLVE, e.g., Redesign Dashboard for New Users]

Sprint Lead: [FACILITATOR NAME]

Team: [LIST NAMES + ROLES]

Stakeholder Sponsor: [NAME + ROLE]

Challenge Statement:

[Write a clear problem to solve. Example: New users feel overwhelmed by the dashboard and don’t understand how to take their first key action.]

Sprint Goal:

[Define success. Example: Define and test a prototype of a simplified onboarding dashboard within 5 days.]

Sprint Timeline:

  • Day 1 – Understand: Map the problem, share insights
  • Day 2 – Diverge: Sketch solutions individually
  • Day 3 – Decide: Vote and select top concepts
  • Day 4 – Prototype: Build clickable prototype
  • Day 5 – Test: Conduct user tests and capture feedback

Daily Structure (Remote):

  • Time: [START TIME] – [END TIME] [TIME ZONE]
  • Platform: [ZOOM / GOOGLE MEET] + [MIRO / FIGMA]
  • Async Prep: [REQUIRED PRE-WORK, e.g., Review research clips]

Roles:

  • Facilitator: [NAME] – Guides process, timekeeping
  • Timekeeper: [NAME] – Alerts on time limits
  • Scribe: [NAME] – Documents decisions in [NOTION LINK]
  • Prototype Builder: [NAME] – Leads Figma build

Tools:

  • Whiteboarding: [MIRO / FIGJAM] – [LINK TO BOARD]
  • Design: [FIGMA] – [LINK TO FILE]
  • Communication: [SLACK CHANNEL] for quick updates

Preparation Required:

  • [REVIEW PRIOR RESEARCH, LINK HERE]
  • [COMPLETE PRE-WORK EXERCISE BY [DATE]]
  • [INSTALL [SOFTWARE] IF NEEDED]

Success Criteria:

  • [PROTOTYPE TESTED WITH 5 USERS BY END OF SPRINT]
  • [3 CLEAR INSIGHTS DOCUMENTED]
  • [NEXT STEPS AGREED FOR ITERATION]

Next Steps:

  • [SEND INVITES TO DAILY SESSIONS]
  • [SHARE ACCESS TO MIRO BOARD]
  • [CONFIRM STAKEHOLDER AVAILABILITY FOR DAY 3 DECISION SESSION]

Template 5: Stakeholder Feedback Brief

When to use: When requesting structured feedback from stakeholders on design concepts.

Project: [PROJECT NAME]

Design Version: [VERSION NUMBER, e.g., v1.2]

Lead Designer: [NAME]

Stakeholders Requested: [LIST NAMES]

Objective of This Review:

[Explain what feedback you need. Example: Confirm visual direction and layout hierarchy for the new homepage.]

What’s Being Reviewed:

  • [PAGE/COMPONENT NAME, e.g., Homepage – Desktop]
  • [FLOW NAME, e.g., Checkout – Step 2]
  • Prototype Link: [LINK]
  • Design File: [FIGMA LINK]

Context:

[Provide background. Example: This design incorporates feedback from usability testing with 8 users. We simplified the navigation and increased font size for readability.]

Key Decisions Made:

  • [DECISION 1, e.g., Used card layout to improve scannability]
  • [DECISION 2, e.g., Moved CTA above the fold]
  • [DECISION 3, e.g., Reduced form fields from 6 to 3]

Specific Questions for Feedback:

  1. [QUESTION 1, e.g., Is the primary CTA prominent enough?]
  2. [QUESTION 2, e.g., Does the layout support user goals?]
  3. [QUESTION 3, e.g., Are there any brand inconsistencies?]

Feedback Guidelines:

  • Focus on user experience and business goals
  • Avoid subjective preferences (e.g., “I don’t like blue”)
  • Reference specific elements using [FIGMA COMMENT TOOL]
  • Submit feedback by: [DATE]

What Happens Next:

  • All feedback compiled by [NAME] on [DATE]
  • Design revisions scheduled for [DATE – DATE]
  • Final version due: [DATE]

Next Steps:

  • [REVIEW DESIGN BY [DATE]]
  • [ADD COMMENTS IN FIGMA]
  • [ATTEND OPTIONAL CLARIFICATION CALL ON [DATE] AT [TIME]]

Template 7: Remote Design Critique Brief

When to use: To structure asynchronous or synchronous design critiques within distributed teams.

Project: [PROJECT NAME]

Design Owner: [DESIGNER NAME]

Critique Lead: [FACILITATOR NAME]

Team Invited: [LIST NAMES]

Design Being Reviewed:

  • [PAGE/FEATURE NAME, e.g., Mobile Profile Editor]
  • Figma Link: [LINK]
  • Prototype: [LINK]

Goal of Critique:

[Explain purpose. Example: Evaluate flow logic and visual clarity before usability testing.]

Context:

[Provide background. Example: This screen allows users to update personal info. It follows new form guidelines from the design system.]

Questions for Reviewers:

  1. [QUESTION 1, e.g., Is the error messaging clear?]
  2. [QUESTION 2, e.g., Does the layout work on small screens?]
  3. [QUESTION 3, e.g., Are required fields obvious?]

Critique Format:

  • [ASYNC: Leave comments in Figma by [DATE]]
  • OR
  • [SYNC: Live session on [DATE] at [TIME] via [ZOOM LINK]]

Feedback Guidelines:

  • Use the “I like, I wish, What if” framework
  • Be specific: reference layers, text, spacing
  • Assume positive intent
  • Avoid personal preferences; focus on user goals

Time Allocation (Live Session):

  • 5 min: Designer presents context
  • 15 min: Silent review (viewing designs)
  • 20 min: Structured feedback round
  • 10 min: Clarifications and next steps

Documenting Feedback:

  • All feedback captured in: [NOTION PAGE / GOOGLE DOC] – [LINK]

Next Steps:

  • [DESIGNER TO REVIEW FEEDBACK BY [DATE]]
  • [UPDATE DESIGN BY [DATE]]
  • [SCHEDULE FOLLOW-UP IF NEEDED]
